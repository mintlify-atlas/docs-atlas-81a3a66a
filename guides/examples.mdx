## Reading and Explaining Code

<Tabs>
  <Tab title="Command">
    ```bash
    marco-code -p "What does the _build_system_prompt function do?"
    ```
  </Tab>
  <Tab title="Tool Calls">
    ```
    [tool] read({'file_path': 'marco_code.py'})
    ```
  </Tab>
  <Tab title="Response">
    The `_build_system_prompt()` function creates the system message that the LLM receives at the start of every conversation. It includes:
    
    1. The assistant's identity and purpose
    2. The current working directory (`os.getcwd()`)
    3. Platform information (`platform.system()` and `platform.release()`)
    
    This context helps the assistant understand the user's environment without requiring them to specify it in prompts.
  </Tab>
</Tabs>

## Running Tests and Fixing Failures

<Tabs>
  <Tab title="Command">
    ```bash
    marco-code -p "Run pytest on test_utils.py and fix any failures"
    ```
  </Tab>
  <Tab title="Tool Calls">
    ```
    [tool] bash({'command': 'pytest test_utils.py -v'})
    [tool] read({'file_path': 'test_utils.py'})
    [tool] read({'file_path': 'utils.py'})
    [tool] write({'file_path': 'utils.py', 'content': '...'})
    [tool] bash({'command': 'pytest test_utils.py -v'})
    ```
    
    The assistant follows a systematic approach:
    1. Run the tests to see what fails
    2. Read the test file to understand expectations
    3. Read the source code
    4. Write corrected code
    5. Re-run tests to verify the fix
  </Tab>
  <Tab title="Response">
    Fixed the bug in `format_date()` - it was returning MM/DD/YYYY instead of YYYY-MM-DD. All tests now pass.
  </Tab>
</Tabs>

## Debugging with Error Context

<Tabs>
  <Tab title="Command">
    ```bash
    marco-code -p "I'm getting KeyError: 'user_id' when running the app. Find and fix it."
    ```
  </Tab>
  <Tab title="Tool Calls">
    ```
    [tool] bash({'command': 'grep -r "user_id" --include="*.py" .'})
    [tool] read({'file_path': './handlers/auth.py'})
    [tool] write({'file_path': './handlers/auth.py', 'content': '...'})
    ```
  </Tab>
  <Tab title="Response">
    Found the issue in `handlers/auth.py:34`. The code was accessing `data['user_id']` directly without checking if the key exists.
    
    Fixed by changing:
    ```python
    user_id = data['user_id']  # KeyError if missing
    ```
    
    To:
    ```python
    user_id = data.get('user_id')
    if not user_id:
        return {"error": "user_id is required"}, 400
    ```
  </Tab>
</Tabs>

## Code Review Tasks

<Tabs>
  <Tab title="Command">
    ```bash
    marco-code -p "Review database.py for security issues and potential SQL injection vulnerabilities"
    ```
  </Tab>
  <Tab title="Tool Calls">
    ```
    [tool] read({'file_path': 'database.py'})
    ```
  </Tab>
  <Tab title="Response">
    Found 2 security concerns:
    
    1. **SQL Injection Risk** in `get_user_by_name()` (line 45):
       ```python
       cursor.execute(f"SELECT * FROM users WHERE name = '{name}'")
       ```
       Use parameterized queries instead:
       ```python
       cursor.execute("SELECT * FROM users WHERE name = ?", (name,))
       ```
    
    2. **Hardcoded credentials** (line 12):
       ```python
       PASSWORD = "admin123"
       ```
       Move to environment variables or a config file not in version control.
  </Tab>
</Tabs>

### Multiple Sequential Tool Calls
```bash
marco-code -p "count Python files and calculate total lines of code"
# [tool] bash({'command': 'find . -name "*.py" | wc -l'})
# [tool] bash({'command': 'find . -name "*.py" | xargs wc -l | tail -1'})
# Response: Found 15 Python files with 3,542 total lines.
```

<Warning>
If the assistant reaches `MAX_ITERATIONS` (10), you'll see:
```
Warning: reached maximum iterations (10)
```

This means the task was too complex for a single invocation. Break it into smaller prompts.
</Warning>

## Tips for Complex Tasks

1. **Be Specific**: "Add type hints to all functions in utils.py" vs "improve utils.py"
2. **Provide Context**: "The tests are failing with AssertionError on line 45" vs "tests are broken"
3. **Check Tool Calls**: Watch stderr to understand the assistant's approach
4. **Iterate if Needed**: If hitting max iterations, continue with a follow-up prompt
